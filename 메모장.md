# 감정분석 연동 설계 메모

## 전체 목표

사용자가 리뷰를 작성하면, 리뷰 텍스트를 FastAPI AI 서버로 보내서 감정 분석(긍정/부정)을 하고
그 결과를 DRF DB에 저장한다.

```
사용자 리뷰 작성
  → DRF (Review 저장)
  → FastAPI (AI 감정 분석)
  → DRF (SentimentResult 저장)
```

---

## 핵심 설계 결정 4가지

### 1. FastAPI 파일 구조 분리

현재 `fastapi/sentiment_test.py` 한 파일에 모든 게 뒤섞여 있음.
역할별로 분리해야 한다.

```
fastapi/
├── config.py    ← MODEL_NAME 등 설정값 (상수)
├── schema.py    ← Pydantic 입출력 모양 (Request / Response)
├── model.py     ← AI 모델 로딩 + 추론 로직
├── router.py    ← API 엔드포인트 (@router.post 들)
├── main.py      ← FastAPI 앱 생성, startup 이벤트, router 등록
└── tests/
    └── test_sentiment.py  ← 테스트 케이스
```

**파일별 역할 요약**

| 파일 | 역할 | 핵심 이유 |
|------|------|-----------|
| config.py | 모델명 등 상수 | 모델 바꿀 때 한 곳만 수정 |
| schema.py | Pydantic 데이터 계약 | DRF와 주고받을 데이터 모양 정의 |
| model.py | AI 모델 관리 | 모델은 서버 시작 시 딱 한 번만 로드해야 함 |
| router.py | 엔드포인트 함수 | main.py가 복잡해지지 않게 분리 |
| main.py | 앱 시작점 | startup 이벤트에서 모델 로드 |
| tests/ | 테스트 | 실제 서버 코드와 분리 |

**중요: 모델은 요청마다 로드하면 안 된다**

```python
# main.py — @app.on_event는 deprecated, lifespan 패턴 사용
@asynccontextmanager
async def lifespan(app: FastAPI):
    load_pipeline()  # 서버 켜질 때 딱 한 번만
    yield

app = FastAPI(lifespan=lifespan)
```

파일 생성 의존 순서: `config → schema → model → router → main → tests`

### 2. DRF - AI 감정 점수 저장 위치

**결론: `SentimentResult` 별도 모델 (OneToOne with Review)**

`Review` 모델에 필드를 바로 붙이지 않는 이유:

- 비동기(Celery)로 처리할 때 `Review.updated_at`이 오염됨
  (리뷰 내용 수정과 AI 점수 갱신이 같은 시각으로 뒤섞임)
- 작업 상태(pending / done / failed) 추적할 곳이 없음
- 재분석 시 `Review` 레코드를 건드리게 됨

```python
# interactions/models.py 에 추가
class SentimentResult(models.Model):
    STATUS_PENDING = "pending"
    STATUS_DONE    = "done"
    STATUS_FAILED  = "failed"

    review      = models.OneToOneField(Review, on_delete=models.CASCADE,
                                       related_name="sentiment")
    status      = models.CharField(max_length=10, default="pending")
    label       = models.CharField(max_length=10, null=True)   # "긍정" / "부정"
    score       = models.FloatField(null=True)                 # 0.0 ~ 1.0
    analyzed_at = models.DateTimeField(null=True)
    error_msg   = models.TextField(blank=True)                 # 실패 시 원인
```

### 3. DRF에서 AI 호출 위치

**Serializer에서 호출하면 안 된다.**

- Serializer의 역할: 데이터 검증 / 변환
- AI 호출 같은 부수효과는 View 레이어 책임

```
AI 호출  → ViewSet.perform_create()
결과 노출 → Serializer (읽기 전용 필드로)
```

```python
# interactions/views.py
def perform_create(self, serializer):
    review = serializer.save(user=self.request.user)
    SentimentResult.objects.create(review=review)         # pending 상태 먼저 생성
    services.request_sentiment(review.id, review.content) # AI 호출
```

```python
# interactions/serializers.py - ReviewSerializer에 읽기 전용 추가
# SentimentResult가 없을 때 null 반환을 위해 SerializerMethodField 사용
sentiment = serializers.SerializerMethodField()

def get_sentiment(self, obj):
    try:
        return SentimentResultSerializer(obj.sentiment).data
    except SentimentResult.DoesNotExist:
        return None
```

### 4. 비동기(Celery) 전환 방향

동기로 먼저 검증 후 Celery로 감싼다.
`services.py` 로직은 그대로 재사용, 호출 방식만 바뀜.

```python
# interactions/tasks.py
@shared_task(bind=True, max_retries=3)
def analyze_sentiment(self, review_id: int, text: str):
    try:
        services.request_sentiment(review_id, text)
    except Exception as exc:
        SentimentResult.objects.filter(review_id=review_id).update(
            status="failed", error_msg=str(exc)
        )
        raise self.retry(exc=exc, countdown=60)
```

```python
# perform_create() 변경 한 줄
# 동기:  services.request_sentiment(review.id, review.content)
# 비동기: tasks.analyze_sentiment.delay(review.id, review.content)
```

## 데이터 흐름 최종 정리

```
[POST /reviews/]
      ↓
ViewSet.perform_create()
      ↓
Review 저장 + SentimentResult(status="pending") 생성
      ↓
Celery 태스크 큐 등록 (analyze_sentiment.delay)
      ↓ (비동기)
tasks.analyze_sentiment 실행
      ↓
services.request_sentiment() → FastAPI POST /analyze
      ↓
FastAPI: AI 모델 추론 → {"label": "긍정", "score": 0.95}
      ↓
SentimentResult 업데이트 (status="done", label, score, analyzed_at)

[GET /reviews/{id}/]
      ↓
ReviewSerializer → sentiment 필드 포함해서 응답
```
