## 🎯 전체 목표

사용자가 리뷰를 작성하면, 리뷰 텍스트를 FastAPI AI 서버로 보내서 감정 분석(긍정/부정)을 하고
그 결과를 DRF DB에 저장한다.

```
사용자 리뷰 작성
  → DRF (Review 저장 + 즉시 응답)
  → Redis (태스크 보관)
  → Celery Worker (백그라운드)
  → FastAPI (AI 감정 분석)
  → DRF DB (SentimentResult 저장)
```

## STEP ① 설계 결정 — 짜기 전에 먼저 결정할 3가지

### 💡 왜 설계를 먼저 하는가?

코드를 바로 짜면 나중에 구조를 갈아엎어야 한다.
특히 AI 추론처럼 **느린 외부 호출**이 포함될 때는
데이터 저장 위치·호출 위치·비동기 여부를 먼저 결정해야
중간에 다 뜯어고치는 일이 없다.

---

### 결정 1 — AI 추론 결과를 어디에 저장할까?

**결론: `Review` 모델이 아닌 `SentimentResult` 별도 모델**

| 문제 | 내용 |
|------|------|
| `updated_at` 오염 | Celery가 점수를 저장하면 `Review.updated_at`이 바뀜 → 리뷰 내용 수정 시각과 AI 분석 시각이 섞임 |
| 상태 추적 불가 | 비동기는 즉시 결과가 안 오므로 pending / done / failed 상태가 필요 |
| 재분석 불가 | 나중에 AI 모델을 바꿔서 재분석할 때 Review 레코드를 건드려야 함 |

---

### 결정 2 — AI는 어디서 호출하는가?

**결론: Serializer가 아닌 ViewSet.perform_create()**

```
Serializer의 역할 → 데이터 검증 / 변환
AI 호출 같은 부수효과 → View 레이어 책임

AI 호출  → ViewSet.perform_create()
결과 노출 → Serializer (읽기 전용 SerializerMethodField)
```

---

### 결정 3 — 동기 vs 비동기

**결론: 처음부터 Celery 비동기**

AI 모델 추론은 수백ms ~ 수초 소요.
동기로 처리하면 리뷰 POST 요청이 AI 응답을 기다리는 동안 블로킹됨.

```
동기:  POST /reviews/ ─────(3초 대기)────→ 200 OK
비동기: POST /reviews/ → 200 OK (즉시)
                      ↓ (백그라운드)
               Celery → AI 호출 → 저장
```

## STEP ② FastAPI 파일 분리

### 💡 왜 하나의 파일을 쪼개는가?

기존 `sentiment_test.py` 한 파일에 5가지 역할이 섞여 있었다.

```python
MODEL_NAME = "..."           # ← 설정값
def load_pipeline(): ...     # ← AI 모델 관리
def to_korean(): ...         # ← 추론 로직
def analyze(): ...           # ← 추론 로직
TEST_CASES = [...]           # ← 테스트 데이터
if __name__ == "__main__":   # ← 테스트 실행
```

### ⚠️ 주의: 패키지 이름 충돌

`fastapi/` 폴더에 `__init__.py`를 넣으면 Python이 설치된 `fastapi` 패키지와 이름이 충돌한다.
→ `__init__.py` 없이 사용하고, `fastapi/` 디렉토리 안에서 `uvicorn main:app` 으로 실행한다.

### 📁 위치

```
fastapi/
├── config.py            ← MODEL_NAME 상수
├── schema.py            ← Pydantic 입출력 모양
├── model.py             ← AI 모델 로딩 + 추론
├── router.py            ← API 엔드포인트
├── main.py              ← FastAPI 앱 생성 + lifespan
├── requirements.txt
└── tests/
    ├── conftest.py      ← pytest 경로 설정
    └── test_sentiment.py
```

파일 생성 의존 순서: `config → schema → model → router → main → tests`

### 📝 schema.py

```python
from pydantic import BaseModel, Field

class SentimentRequest(BaseModel):
    text: str = Field(..., min_length=1, max_length=5000, description="분석할 텍스트")

class SentimentResponse(BaseModel):
    label: str   = Field(..., description="긍정 / 부정")
    score: float = Field(..., ge=0.0, le=1.0, description="신뢰도 (0.0 ~ 1.0)")
```

### 📝 model.py

```python
import logging
from transformers import pipeline, Pipeline
from config import MODEL_NAME

logger = logging.getLogger(__name__)
_clf: Pipeline | None = None


def load_pipeline() -> Pipeline:
    global _clf
    _clf = pipeline("text-classification", model=MODEL_NAME)
    return _clf


def get_pipeline() -> Pipeline:
    if _clf is None:
        raise RuntimeError("모델이 로드되지 않았습니다. lifespan 설정을 확인하세요.")
    return _clf


def predict(text: str) -> dict:
    clf = get_pipeline()
    result = clf(text)[0]
    # LABEL_1 = 긍정, LABEL_0 = 부정
    label = "긍정" if "1" in result["label"] else "부정"
    return {"label": label, "score": round(float(result["score"]), 4)}
```

### 📝 router.py

```python
from fastapi import APIRouter, HTTPException
from schema import SentimentRequest, SentimentResponse
from model import predict

router = APIRouter()

@router.post("/analyze", response_model=SentimentResponse, summary="감성 분석")
def analyze(req: SentimentRequest) -> SentimentResponse:
    try:
        result = predict(req.text)
    except RuntimeError as exc:
        raise HTTPException(status_code=503, detail=str(exc))
    return SentimentResponse(**result)

@router.get("/health", summary="헬스 체크")
def health() -> dict:
    return {"status": "ok"}
```

### 📝 main.py

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from model import load_pipeline
from router import router

@asynccontextmanager
async def lifespan(app: FastAPI):
    load_pipeline()  # 서버 시작 시 딱 한 번만 로드
    yield

app = FastAPI(title="감성 분석 API", version="1.0.0", lifespan=lifespan)
app.include_router(router)
```

### 🔑 핵심 포인트

| 코드 | 이유 |
|------|------|
| `lifespan` 패턴 | `@app.on_event("startup")`은 deprecated. `lifespan`이 최신 방식 |
| `global _clf` | 모델을 모듈 수준 변수에 보관 → 요청마다 재로드 방지 |
| `"1" in result["label"]` | 모델이 `LABEL_1` 형태로 반환하므로 `== "1"` 이 아닌 `in` 으로 체크 |
| `503` (RuntimeError) | 모델 미로드 상태는 서버 오류이므로 503 반환 |

## STEP ③ DRF 모델 추가 — SentimentResult

### 📝 ai/ai_sentiment/models.py

```python
class SentimentResult(models.Model):
    STATUS_PENDING = "pending"
    STATUS_DONE    = "done"
    STATUS_FAILED  = "failed"
    STATUS_CHOICES = [
        (STATUS_PENDING, "분석 대기"),
        (STATUS_DONE,    "분석 완료"),
        (STATUS_FAILED,  "분석 실패"),
    ]

    review = models.OneToOneField(
        "interactions.Review",
        on_delete=models.CASCADE,
        related_name="sentiment",
    )
    status      = models.CharField(max_length=10, choices=STATUS_CHOICES,
                                   default=STATUS_PENDING, db_index=True)
    label       = models.CharField(max_length=10, null=True, blank=True)
    score       = models.FloatField(null=True, blank=True)
    analyzed_at = models.DateTimeField(null=True, blank=True)
    error_msg   = models.TextField(blank=True)

    @classmethod
    def reset_for_review(cls, review_id: int) -> None:
        """리뷰 내용 수정 시 이전 분석 결과를 초기화하고 pending으로 되돌린다."""
        cls.objects.filter(review_id=review_id).update(
            status=cls.STATUS_PENDING,
            label=None, score=None, analyzed_at=None, error_msg="",
        )

    class Meta:
        db_table = "sentiment_result"
```

### migration 실행

```bash
python manage.py makemigrations interactions --name add_sentiment_result
python manage.py migrate
```

### 🔑 핵심 포인트

| 코드 | 이유 |
|------|------|
| `OneToOneField` | 리뷰 1개 = 감성 결과 1개 |
| `"interactions.Review"` 문자열 참조 | ai → interactions 순환 import 방지 |
| `db_index=True` (status) | "pending인 것만 재처리" 쿼리 성능 향상 |
| `reset_for_review()` classmethod | 리셋 로직을 모델에 캡슐화 → 여러 곳에서 재사용 |

## STEP ④ DRF 서비스 작성 — FastAPI 호출

### 📝 ai/ai_sentiment/services.py

```python
import logging
import requests
from django.conf import settings
from django.utils import timezone
from .models import SentimentResult

logger = logging.getLogger(__name__)


def request_sentiment(review_id: int, text: str) -> None:
    url     = settings.FASTAPI_SENTIMENT_URL
    timeout = settings.FASTAPI_SENTIMENT_TIMEOUT
    try:
        resp = requests.post(url, json={"text": text}, timeout=timeout)
        resp.raise_for_status()
        data = resp.json()
        SentimentResult.objects.filter(review_id=review_id).update(
            status=SentimentResult.STATUS_DONE,
            label=data["label"],
            score=data["score"],
            analyzed_at=timezone.now(),
            error_msg="",
        )
    except Exception as exc:
        logger.error("감성 분석 실패 review_id=%s: %s", review_id, exc)
        SentimentResult.objects.filter(review_id=review_id).update(
            status=SentimentResult.STATUS_FAILED,
            error_msg=str(exc),
        )
        raise
```

## STEP ⑤ 시리얼라이저 수정 — sentiment 읽기 필드 추가

### 💡 왜 SerializerMethodField인가?

`SentimentResult`가 없는 리뷰에서 `review.sentiment`에 접근하면
`RelatedObjectDoesNotExist` 예외가 발생한다.
중첩 Serializer는 이 예외를 자동으로 처리하지 못하므로
`SerializerMethodField`로 try/except 처리한다.

### 📝 interactions/serializers.py

```python
from ai.ai_sentiment.models import SentimentResult
from ai.ai_sentiment.serializers import SentimentResultSerializer

class ReviewSerializer(serializers.ModelSerializer):
    sentiment = serializers.SerializerMethodField()

    def get_sentiment(self, obj):
        try:
            return SentimentResultSerializer(obj.sentiment).data
        except SentimentResult.DoesNotExist:
            return None
```

### 리뷰 GET 응답 예시

```json
{
  "content": "음식이 정말 맛있었어요!",
  "sentiment": {
    "status": "done",
    "label": "긍정",
    "score": 0.9823,
    "analyzed_at": "2026-05-21T10:30:00+09:00"
  }
}
```


## STEP ⑥ 뷰 수정 — perform_create에서 태스크 호출

### 📝 interactions/views.py — ReviewViewSet 핵심 수정

```python
from ai.ai_sentiment.models import SentimentResult
from ai.ai_sentiment import tasks

class ReviewViewSet(viewsets.ModelViewSet):

    def get_queryset(self):
        # sentiment까지 한 번에 조인 — N+1 방지
        qs = Review.objects.select_related("restaurant", "user", "sentiment")
        ...

    def perform_create(self, serializer):
        review = serializer.save(user=self.request.user)
        SentimentResult.objects.create(review=review)           # pending 먼저 생성
        tasks.analyze_sentiment.delay(review.id, review.content)  # 비동기 등록

    def update(self, request, *args, **kwargs):
        instance = self.get_object()
        if instance.user != request.user:
            raise PermissionDenied("권한이 없습니다.")
        serializer = self.get_serializer(instance, data=request.data, partial=True)
        serializer.is_valid(raise_exception=True)
        self.perform_update(serializer)

        if "content" in serializer.validated_data:  # 내용 변경 시에만 재분석
            SentimentResult.reset_for_review(instance.id)
            tasks.analyze_sentiment.delay(instance.id, instance.content)
        return Response(serializer.data)

    def perform_destroy(self, instance):
        if instance.user != self.request.user:
            raise PermissionDenied("권한이 없습니다.")
        instance.delete()
```

### 🔑 핵심 포인트

| 코드 | 이유 |
|------|------|
| `select_related("sentiment")` | sentiment 접근마다 쿼리 N번 발생하는 N+1 버그 방지 |
| `SentimentResult.objects.create()` | pending 먼저 생성 → 클라이언트가 null 대신 pending 응답 가능 |
| `tasks.analyze_sentiment.delay()` | `.delay()`가 Celery 비동기 호출 방법 |
| `perform_destroy()` 오버라이드 | `destroy()`에서 `get_object()` 이중 호출(DB 쿼리 2번) 제거 |

## STEP ⑦ Celery 비동기 전환

### 💡 서버 3개의 역할 — 식당 비유

```
손님 (브라우저)
    ↓
홀 직원   — DRF 서버 (:8000)   리뷰 저장, 즉시 응답
    ↓
전표 창고  — Redis              "분석해줘" 쪽지 보관
    ↓
주방 직원  — Celery Worker      쪽지 꺼내서 FastAPI 호출
    ↓
전문 요리사 — FastAPI (:8001)   AI 모델로 긍정/부정 분석
```

| 서버 | 없으면 |
|------|--------|
| DRF | 사이트 자체가 안 열림 |
| Celery | 리뷰 써도 AI 분석 안 됨 (pending 그대로) |
| FastAPI | Celery가 호출해도 분석 결과 없음 (failed) |

### 📝 mysite/celery.py

```python
import os
from celery import Celery

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "mysite.settings")
app = Celery("mysite")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()  # INSTALLED_APPS 훑으며 tasks.py 자동 등록
```

### 📝 mysite/__init__.py

```python
from .celery import app as celery_app
__all__ = ["celery_app"]
```

> ⚠️ `__init__.py`에 등록하지 않으면 Django 시작 시 Celery 앱이 로드되지 않아
> `@shared_task`가 제대로 작동하지 않는다.

### 📝 ai/ai_sentiment/tasks.py

```python
from celery import shared_task
from . import services

@shared_task(bind=True, max_retries=3, default_retry_delay=60)
def analyze_sentiment(self, review_id: int, text: str) -> None:
    """실패 시 최대 3회 60초 간격 재시도. 로깅은 services.py에서 처리."""
    try:
        services.request_sentiment(review_id, text)
    except Exception as exc:
        raise self.retry(exc=exc)
```

### 🔑 핵심 포인트

| 코드 | 이유 |
|------|------|
| `@shared_task` | 앱 인스턴스에 직접 묶이지 않아 재사용 가능 |
| `bind=True` | `self` 파라미터로 재시도 등 태스크 메타 접근 가능 |
| `max_retries=3` | FastAPI 서버 일시 장애 시 자동 재시도 |
| `default_retry_delay=60` | 재시도 간격 60초 — 즉시 재시도는 의미 없음 |
| `autodiscover_tasks()` | 새 ai 앱 추가 시 celery.py 수정 없이 자동 인식 |

## STEP ⑧ 리팩토링 — 발견된 문제 7가지

### 문제 1 — N+1 쿼리 (버그)


### 해결: SentimentDisplayMixin + 상수 추출

```python
# ai/ai_sentiment/admin.py

# 모듈 레벨 상수 — interactions/admin.py에서도 import해서 사용
STATUS_STYLE = {
    SentimentResult.STATUS_DONE:    ("#d4edda", "#155724", "분석 완료"),
    SentimentResult.STATUS_PENDING: ("#fff3cd", "#856404", "분석 대기"),
    SentimentResult.STATUS_FAILED:  ("#f8d7da", "#721c24", "분석 실패"),
}
LABEL_STYLE = {
    "긍정": ("#d4edda", "#155724"),
    "부정": ("#f8d7da", "#721c24"),
}
DEFAULT_BADGE = ("#e9ecef", "#495057")


def _badge(bg, color, label, weight="600"):
    return format_html(
        '<span style="background:{};color:{};padding:3px 10px;'
        'border-radius:999px;font-size:12px;font-weight:{};">{}</span>',
        bg, color, weight, label,
    )


class SentimentDisplayMixin:
    """status_badge, label_badge, score_bar — Inline과 Admin 공용."""

    @admin.display(description="분석 상태")
    def status_badge(self, obj):
        bg, color, label = STATUS_STYLE.get(obj.status, (*DEFAULT_BADGE, obj.status))
        return _badge(bg, color, label)

    @admin.display(description="감정")
    def label_badge(self, obj):
        if not obj.label:
            return "—"
        bg, color = LABEL_STYLE.get(obj.label, DEFAULT_BADGE)
        return _badge(bg, color, obj.label, weight="700")

    @admin.display(description="신뢰도")
    def score_bar(self, obj):
        if obj.score is None:
            return "—"
        pct       = int(obj.score * 100)
        score_txt = f"{obj.score * 100:.1f}%"
        color     = "#28a745" if obj.label == "긍정" else "#dc3545"
        return format_html(
            '<div style="display:flex;align-items:center;gap:8px;">'
            '<div style="width:120px;background:#e9ecef;border-radius:4px;height:10px;">'
            '<div style="width:{}%;background:{};border-radius:4px;height:10px;"></div>'
            '</div><span style="font-size:13px;font-weight:600;">{}</span></div>',
            pct, color, score_txt,
        )


class SentimentResultInline(SentimentDisplayMixin, admin.StackedInline):
    model = SentimentResult
    ...

class SentimentResultAdmin(SentimentDisplayMixin, admin.ModelAdmin):
    ...
```

## STEP ⑬ FastAPI 구조 개선 — models/, routers/ 디렉토리 분리

### 💡 왜 분리하는가?

AI 기능이 감성분석 하나일 때는 `model.py`, `router.py` 단일 파일로 충분했다.
가짜 리뷰 탐지가 추가되면서 동일한 이름의 파일에 두 역할이 섞이게 된다.
역할별 디렉토리로 분리하면 파일이 늘어나도 어디에 무엇이 있는지 명확하다.

### 변경 내용

| 전 (단일 파일) | 후 (디렉토리) |
|----------------|---------------|
| `fastapi/model.py` | `fastapi/models/sentiment.py` |
| `fastapi/router.py` | `fastapi/routers/sentiment.py` |
| _(없음)_ | `fastapi/models/fake_review.py` |
| _(없음)_ | `fastapi/routers/fake_review.py` |

### 📝 fastapi/main.py (최종)

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from models.sentiment   import load_pipeline as load_sentiment_model
from models.fake_review import load_models   as load_fake_review_models
from routers            import sentiment, fake_review

@asynccontextmanager
async def lifespan(app: FastAPI):
    load_sentiment_model()      # koelectra 감성분석 모델
    load_fake_review_models()   # Helsinki-NLP 번역 모델 + 가짜탐지 모델
    yield

app = FastAPI(title="AI 분석 API", version="2.0.0", lifespan=lifespan)
app.include_router(sentiment.router)    # /analyze, /health
app.include_router(fake_review.router)  # /fake-review/detect
```

### 🔑 핵심 포인트

| 코드 | 이유 |
|------|------|
| `load_pipeline as load_sentiment_model` | 두 함수 이름이 충돌하지 않게 alias 사용 |
| lifespan에서 두 모델 순서 로드 | 감성분석 먼저 → 번역+탐지 순서, 한 번만 로드 |
| `models/__init__.py` 빈 파일 | Python이 디렉토리를 패키지로 인식하게 함 |

## STEP ⑭ 가짜 리뷰 탐지 AI — 번역 + 탐지 파이프라인

### 💡 왜 번역이 필요한가?

가짜 리뷰 탐지 모델은 **영어 전용**으로 학습되어 있다.
한국어 리뷰를 직접 넣으면 정확한 분석이 안 된다.
한국어 → 영어 번역 후 탐지 모델에 넣어야 한다.

### 번역 방법 선택 과정

| 방법 | 비용 | 결정 |
|------|------|------|
| Papago (Naver NCP) | 유료, 무료 티어 없음 | ❌ 제외 |
| DeepL | 월 50만자 무료 | 검토했으나 API 키 발급 필요 |
| Helsinki-NLP/opus-mt-ko-en | 완전 무료, 로컬 실행 | ✅ 채택 |

### 전체 흐름

```
한국어 리뷰
    ↓
[FastAPI] Helsinki-NLP 번역 (한국어 → 영어)
    ↓
[FastAPI] mrm8488/bert-tiny-finetuned-sms-spam-detection 탐지
    ↓
{ is_fake: true/false, confidence: 0.93, translated_text: "..." }
    ↓
[DRF] FakeReviewResult 저장 + 패널티 적용
```

### 가짜 탐지 모델 선택 과정

| 모델 | 문제 | 결정 |
|------|------|------|
| `zayuki/computer_generated_fake_review_detection` | 가중치 파일(model.safetensors) 없음 — 불완전 업로드 | ❌ 사용 불가 |
| `mrm8488/bert-tiny-finetuned-sms-spam-detection` | 17.6MB, 정확도 98%, 영어 스팸/가짜 탐지 | ✅ 채택 |

### 📝 fastapi/models/fake_review.py

```python
import torch
from transformers import AutoModelForSeq2SeqLM, AutoTokenizer, pipeline, Pipeline
from config import FAKE_REVIEW_MODEL_NAME, TRANSLATION_MODEL_NAME

_tokenizer:   AutoTokenizer          | None = None
_trans_model: AutoModelForSeq2SeqLM  | None = None
_detector:    Pipeline               | None = None


def load_models() -> None:
    global _tokenizer, _trans_model, _detector
    _tokenizer   = AutoTokenizer.from_pretrained(TRANSLATION_MODEL_NAME)
    _trans_model = AutoModelForSeq2SeqLM.from_pretrained(TRANSLATION_MODEL_NAME)
    _detector    = pipeline("text-classification", model=FAKE_REVIEW_MODEL_NAME)


def _translate_ko_to_en(text: str) -> str:
    inputs = _tokenizer(text, return_tensors="pt", padding=True,
                        truncation=True, max_length=512)
    with torch.no_grad():
        outputs = _trans_model.generate(**inputs, max_length=512)
    return _tokenizer.decode(outputs[0], skip_special_tokens=True)


def detect(korean_text: str) -> dict:
    english_text = _translate_ko_to_en(korean_text)
    result       = _detector(english_text[:512])[0]
    is_fake      = result["label"].upper() in ("FAKE", "LABEL_1", "1")
    confidence   = round(float(result["score"]), 4)
    return {
        "is_fake":         is_fake,
        "confidence":      confidence,
        "translated_text": english_text,
    }
```

```python
def request_fake_review_check(review_id: int, korean_text: str) -> None:
    resp = requests.post(
        settings.FASTAPI_FAKE_REVIEW_URL,
        json={"text": korean_text},
        timeout=settings.FASTAPI_FAKE_REVIEW_TIMEOUT,  # settings에서 관리
    )
    resp.raise_for_status()
    data = resp.json()

    is_penalized = data["is_fake"] and data["confidence"] >= PENALTY_THRESHOLD
    penalty      = FakeReviewResult.PENALTY_FAKE if is_penalized else 0

    FakeReviewResult.objects.filter(review_id=review_id).update(
        status=FakeReviewResult.STATUS_DONE,
        is_fake=data["is_fake"],
        confidence=data["confidence"],
        translated_text=data["translated_text"],
        penalty_score=penalty,
        analyzed_at=timezone.now(),
        error_msg="",
    )

    if is_penalized:
        _apply_penalty(review_id)


def _apply_penalty(review_id: int) -> None:
    from interactions.models import Review
    from accounts.models import UserTrustScore

    review = Review.objects.select_related("user").get(id=review_id)
    UserTrustScore.objects.filter(user=review.user).update(
        score      = F("score") + FakeReviewResult.PENALTY_FAKE,
        fake_count = F("fake_count") + 1,
    )
```

> 💬 **핵심 교훈 13가지**
>
> 1. **설계는 코드 전에** — 저장 위치·호출 위치·동기/비동기를 먼저 결정해야 중간에 갈아엎는 일이 없다
> 2. **Serializer에 부수효과 금지** — AI 호출·메일 발송 등 사이드이펙트는 View 레이어에서 처리한다
> 3. **`QuerySet.update()`로 부분 업데이트** — `instance.save()`는 `auto_now` 필드까지 갱신하므로 AI 결과 저장 시는 `.update()` 사용
> 4. **`select_related` 누락은 조용한 N+1** — 에러 없이 느려지기만 하므로 관련 모델 조회 시 반드시 확인
> 5. **로직을 모델 classmethod로 캡슐화** — `reset_for_review()`처럼 여러 곳에서 쓰이는 로직은 모델 안으로
> 6. **중복 로깅은 원인 찾기를 어렵게 한다** — 레이어를 명확히 나누고 로깅은 책임 있는 한 곳에서만
> 7. **모델 이동 시 SeparateDatabaseAndState** — 데이터 보존하며 앱 간 모델 이동 가능, DB 재생성 불필요
> 8. **Celery는 프로젝트 당 1개** — 각 ai 앱은 `@shared_task`로 태스크만 등록, `autodiscover_tasks()`가 자동 수집
> 9. **HuggingFace 모델 사용 전 가중치 파일 확인** — model.safetensors 없는 모델은 로드 자체가 실패한다. 페이지에서 Files 탭 먼저 확인
> 10. **번역이 필요한 영어 전용 모델** — 한국어 입력 시 무조건 번역 후 추론, 번역 결과를 DB에 저장해 탐지 근거 보존
> 11. **패널티는 임계값(threshold) 설정 필수** — 신뢰도 낮은 탐지 결과로 패널티를 주면 오탐이 발생. 85% 이상만 적용
> 12. **Admin Mixin으로 표시 메서드 재사용** — `_badge()`, `score_bar()` 같은 HTML 렌더링 메서드는 Mixin으로 추출해 Inline/Admin 양쪽 공용 사용
> 13. **settings에서 모든 타임아웃 관리** — 외부 서비스 타임아웃 하드코딩 시 운영 환경에서 수정 불가, 반드시 env 변수로
